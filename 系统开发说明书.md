## 2026-04-09 开发记录（补充）

### 2. 修复登录认证问题 - Token 认证机制完善

**问题**: 组织机构管理新增时提示"无效的 Token"，所有表单增删改查操作失败

**问题原因**:
1. **前端使用模拟登录**: `Login.vue` 组件没有调用真实后端 API，而是使用 `userStore.login('mock-token', ...)` 模拟登录
2. **Token 无效**: 存储的 `'mock-token'` 不是有效的 JWT Token，后端无法验证
3. **认证失败**: 所有需要 `@jwt_required()` 验证的 API 请求都返回 401 UNAUTHORIZED

**Token 认证机制说明**:
- **用户权限认证**: 验证用户身份和角色（admin, user）
- **API 接口认证**: 所有增删改查操作都需要有效的 JWT Token
- **双重保护**: 既要验证用户权限，也要验证 API 访问令牌

**解决方案**:
1. 修改 `Login.vue` 调用真实后端登录 API `/api/auth/login`
2. 存储后端返回的 `access_token` 到 localStorage
3. 前端 API 请求使用存储的 Token 进行认证
4. 统一 Token 键名为 `token`

**修改文件**:
- `frontend/src/views/auth/Login.vue` - 登录页面（核心修复）
- `frontend/src/views/admin/Organization.vue` - 组织机构管理
- `frontend/src/views/admin/UserManagement.vue` - 用户管理
- `frontend/src/views/admin/UserApproval.vue` - 用户审核
- `frontend/src/views/admin/Role.vue` - 角色管理
- `frontend/src/views/Home.vue` - 首页
- `frontend/src/components/Layout.vue` - 布局组件

**技术实现**:
```javascript
// 登录时调用真实 API
async handleLogin() {
  const response = await fetch('/api/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password })
  })
  
  const result = await response.json()
  
  if (result.code === 200) {
    // 存储后端返回的 access_token
    userStore.login(result.data.access_token, result.data.user)
  }
}

// API 请求时携带 Token
const response = await fetch('/api/organization', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${localStorage.getItem('token')}`
  },
  body: JSON.stringify(formData)
})
```

**影响范围**:
- ✅ 所有需要认证的表单新增功能
- ✅ 所有需要认证的表单编辑功能
- ✅ 所有需要认证的表单删除功能
- ✅ 所有需要认证的列表查询功能
- ✅ 组织机构管理、用户管理、用户审核、角色管理等所有模块

### 2026-04-09 开发记录（补充：登录失败与 Redis 引用修复）

#### 1. 管理员登录提示“请检查网络连接”（实际为后端 500）

**现象**:
- 登录页输入 `admin/admin123` 后前端提示“登录失败，请检查网络连接”
- 浏览器控制台显示 `POST /api/auth/login 500 (Internal Server Error)`

**根因分析**:
- `exam_system/api/auth.py` 采用 `from exam_system.extensions import redis_client` 在模块导入时绑定变量
- 应用启动时 `init_redis(app)` 才会初始化 `extensions.redis_client`，但 `auth.py` 中的 `redis_client` 仍为旧值（可能为 `None`）
- 导致调用 `redis_client.exists/get/setex` 抛异常，触发 500，前端解析失败后误认为网络问题

**解决方案**:
- `auth.py` 改为动态引用 `extensions.redis_client`，避免导入绑定导致的空引用问题
- `extensions.py` 在 Redis 连接失败时使用进程内缓存实现，保证登录失败计数逻辑可用（避免接口崩溃）
- `Login.vue` 使用统一的 `authApi.login`（axios 实例 `/api`），并优先展示后端返回的 `message`

**修改文件**:
- `exam_system/api/auth.py` - 动态获取 `extensions.redis_client`
- `exam_system/extensions.py` - Redis 不可用时启用内存缓存实现
- `frontend/src/views/auth/Login.vue` - 登录请求与错误提示优化

### 2026-04-10 开发记录（组织机构 401/闪退、审批与数据管理模块）

#### 1. JWT 身份与管理员接口

**现象**：首页与布局中请求组织机构、用户审批接口出现 401，或进入组织机构管理后 axios 全局拦截导致退回登录页。

**处理**：
- 新增 `exam_system/utils/jwt_helper.py` 中 `resolve_user_id()`，统一将 `get_jwt_identity()` 转为整型再 `User.query.get`，避免类型不一致导致权限判断异常。
- `organization`、`user_management`、`decorators` 中管理员校验改为使用 `resolve_user_id()`。
- `docker-compose.yml` 为后端增加固定 `JWT_SECRET_KEY`，避免容器重建导致旧 Token 全部失效。
- 首页仅管理员请求组织机构信息；布局中未读数仅管理员请求 `pending`，避免非管理员无意义请求。
- axios 响应拦截：登录/注册接口返回 401（如密码错误）时不触发全局退出。

#### 2. 用户注册审批与「已读」数据流

- 后端新增 `GET /api/user-management/approved`：已审核通过且非管理员用户列表，供审批页「已读消息」分类使用（原 `/approved` 缺失导致 404）。
- 注册成功写入 `biz_operation_logs`（操作状态「提交」）；审核通过写入业务日志（「审核」）。
- 前端 `UserApproval.vue`、`UserManagement.vue` 统一改用 `userManagementApi`（axios + Token）。

#### 3. 数据管理：在线用户、日志、备份、恢复

- 新表：`online_users`（登录写入、登出删除）、`biz_operation_logs`（业务日志）。
- 新接口：`/api/online-users`、`/api/biz-operation-logs`、`/api/data-backup`、`/api/data-restore`（MySQL 逻辑备份为 `.sql`；恢复后建议重启后端）。
- 前端：`DataLayout.vue` 子路由 `online-users`、`logs`、`backup`、`restore`；`Layout.vue` 数据管理下增加三级子菜单项。

### 2026-04-10 前端会话与管理员权限（401 闪退、审批图标）

**问题**：首页/布局请求组织机构、待审核列表返回 401 时，axios 全局退出；非管理员误进后台菜单。

**处理**：

- `/system` 路由增加 `meta.requiresAdmin`，守卫中校验 `userInfo.role === 'admin'`，否则跳转首页并提示。
- `Layout.vue`：系统设置整块菜单、头部消息角标仅 `isAdmin` 显示。
- `UserApproval.vue`：未读/已读/已删除分类使用 `msg-unread.svg`、`msg-read.svg`、`msg-deleted.svg` 图标资源。

### 2026-04-10 部署后 403 无法登录

**原因**：（1）`users` 表中 `admin` 为待审核 `status=0` 时，`/api/auth/login` 返回 HTTP 403；（2）宿主 `frontend/dist` 权限过严时，Nginx 进程无法读静态文件，浏览器报 403。

**处理**：`init_admin_user()` 在已存在 `admin` 用户且 `status==0` 时改为 `1`（不自动解除 `status==2` 禁用）；部署前对 `frontend/dist` 使用 `chmod -R a+rX`，避免 Nginx worker 非 root 时无法读静态资源。
